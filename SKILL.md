# Deck Studio Engine — 咨询级 PPT 全流程生产引擎

## 设计哲学

> **对平庸零容忍**——不是口号，是写在代码里的质量门禁。

本引擎源于对「打磨咨询级 PPT 技能」全过程的深度复盘。它不是"能跑 demo"的半成品，而是经过五轮实测、26 项端到端测试、家族公约审计的"可交付产品"。

**核心理念：**"能跑"到"能用"的距离远比想象的长——模板标准化、合规预检、并行渲染、缓存机制、智能主题推荐，每一步都是让输出从"AI 味儿"走向"专业感"的关键。

## 1. 触发条件

当用户表达以下意图时触发本技能：

| 触发类型 | 示例 |
|---------|------|
| 一句话需求 | "帮我做一份关于 XX 的 PPT" / "生成一份 AI 行业研究报告" |
| 多页文档生成 | "写一份 15 页的 AI 创业 BP" |
| 特定场景 | "路演融资 PPT" / "技术发布会" / "政策解读" / "产品发布" |
| 素材转化 | "把这篇研究报告转成 PPT" / "根据这个大纲做 PPT" |
| 快速迭代 | "现有的 PPT 质量不够好，重新做一版" |

**默认输出：** 一份可直接面向决策者的 `.pptx` 文件。

## 2. 五阶段流水线

整个生产过程分为五个阶段，**每阶段都有质量门禁**——不达标则打回重做。

    ▼

## 3. 工作流详细规范

### 3.1 DeepScout — 深度研究

**输入：** 用户的一句话需求
**输出：** `output/research_briefing.md`

**行为规范：**
1. 先拆解需求中的核心主题、目标受众、场景类型
2. 多源数据采集（互联网搜索 + 本地知识库）：
   - 行业数据（市场规模、增长率、竞争格局）
   - 政策法规（最新政策、行业标准）
   - 技术趋势（前沿技术、专利布局）
   - 标杆案例（龙头企业、最佳实践）
3. 框架驱动填充：按预设的信息架构模板填充内容
4. 交叉验证数据来源，标记不确定信息

**质量门禁：**
- [ ] 数据覆盖度：每个关键维度至少 2 个独立来源
- [ ] 时效性：数据时间标记明确，优先使用最新数据
- [ ] 准确性：无事实性错误，不确定内容需标注

### 3.2 Architect — 叙事架构

**输入：** `research_briefing.md`
**输出：** `output/narrative_structure.md`

**行为规范：**
1. 提炼"灵魂句"（soul sentence）——整份 PPT 的核心主张，一句话说清"为什么听你的"
2. 构建叙事骨架，使用 MECE（Mutually Exclusive, Collectively Exhaustive）原则验证
3. 设计认知弧线：开场共鸣 → 问题呈现 → 方案展开 → 数据佐证 → 结论升华
4. 为每页分配：标题、核心信息、数据支撑要点

**标准叙事结构模板：**
```
1. 封面 + 灵魂句
2. 目录/议程
3. 背景/现状分析（2-3 页）
4. 核心问题/痛点（2-3 页）
5. 方案/解决思路（3-5 页）
6. 数据/案例佐证（2-3 页）
7. 实施路径/路线图（1-2 页）
8. 价值/ROI（1-2 页）
9. 总结/行动呼吁（1 页）
```

**质量门禁：**
- [ ] 灵魂句是否清晰、有力、可记忆
- [ ] 叙事骨架是否通过 MECE 验证
- [ ] 认知弧线是否有足够的张力（起承转合）
- [ ] 每页是否有明确的信息目标

### 3.3 Visual Director — 视觉导演

**输入：** `narrative_structure.md`
**输出：** `output/visual_blueprint.md`

**行为规范：**
1. 为每页选择**视觉主角类型**（参考 §5）
2. 运行**五维度智能主题推荐**（参考 §6），选择最匹配的品牌预设
3. 为每页分配布局模板（参考 §A 模板蓝图）
4. 指定图表需求（参考 §B 图表类型）
5. 定义信息密度等级：低(留白) / 中(平衡) / 高(数据密集)

**视觉主角选择规则：**
- 数据分析页 → 数据可视化型
- 方案介绍页 → 概念示意型
- 案例展示页 → 大图故事型
- 对比论证页 → 对比拆分型
- 时间线/路线图 → 流程叙事型
- 封面/总结页 → 情感共鸣型

**质量门禁：**
- [ ] 每页的视觉主角类型与内容语义匹配
- [ ] 品牌主题与场景匹配（路演场景用 pitch-vc 主题等）
- [ ] 信息密度与页面类型一致（数据报告不应用叙事型主题）

### 3.4 SlideWright — 幻灯片渲染

**输入：** `visual_blueprint.md`
**输出：** `output/deck.pptx`

**行为规范：**

#### 3.4.1 合规预检（渲染前飞行检查）

在渲染之前，必须执行以下检查：
1. **数据完整性**：Blueprint 中每页是否有完整的内容数据和元数据
2. **格式规范性**：模板类型是否在 12 种布局家族中，图表类型是否在 16 种中
3. **品牌一致性**：所选品牌主题的颜色、字体、间距是否符合预设

**检查失败 → 返回 Visual Director 修正，不进入渲染流程。**

#### 3.4.2 确定性渲染路径

- 从 Blueprint 到 SVG 到 PPTX 的路径必须是**确定性的**——相同输入始终产生相同输出
- 不依赖 AI 在渲染阶段的"临场发挥"
- 使用 Blueprint 规范中的槽位映射规则进行精确内容绑定

#### 3.4.3 并行渲染

- **页面之间没有依赖关系，必须并行渲染**
- 使用多进程/多线程并行处理各页面的 SVG 生成
- 然后合并为最终的 PPTX 文件

⚠️ **Windows 兼容性警告：** 如果使用 `ProcessPoolExecutor`，必须在 `if __name__ == '__main__':` 保护下创建，否则 Windows 上子进程会递归创建导致卡死。

#### 3.4.4 缓存机制

- **相同布局结构 + 相同数据类型 = 相同渲染结果**
- 计算过的结果缓存起来，下次直接复用，不做重复计算
- 缓存键：`{layout_type}:{data_type_signature}`

#### 3.4.5 编码规范

⚠️ **Windows GBK 编码注意：** Windows 默认编码是 GBK。读写文件时必须显式指定编码：
- 输出文件用 `encoding="utf-8"`（PPTX 内部使用 UTF-8）
- 当读取 Windows 本地文本文件时，注意编码兼容：`open(file, encoding="utf-8", errors="replace")`

### 3.5 Compass — 质量审查

**输入：** `deck.pptx` + `visual_blueprint.md`
**输出：** `output/final_review_report.md`

**三重审查机制：**

| 级别 | 审查内容 | 标准 |
|------|---------|------|
| L1 逻辑审查 | 叙事是否完整、逻辑链是否断裂、结论是否有支撑 | 检查灵魂句是否贯穿全文 |
| L2 数据审查 | 数据准确性、图表与数据一致性、来源可追溯 | 对比研究简报和幻灯片中的数据 |
| L3 视觉审查 | 配色、排版、字体一致性、信息密度合理性 | 对照品牌预设规范 |

**审查结论：**
- ✅ 通过 → 交付
- ⚠️ 小问题 → 标记建议，交付但不阻塞
- ❌ 不通过 → 打回对应阶段修复

## 4. 工程规范（家族公约）

技能文件结构和版本管理必须遵守以下规范：

### 4.1 文件结构标准
```
deck-studio-engine/
├── SKILL.md              # 本文件
├── README.md             # 品牌标识 + 使用说明（版本号与实际一致）
├── CHANGELOG.md          # 变更日志（## [x.y.z] - date 格式）
├── references/           # 引用资源（设计规范、模板、示例）
│   ├── templates-blueprint.md
│   ├── brand-themes.yaml
│   ├── chart-specs.md
│   ├── visual-protagonists.md
│   ├── theme-scoring.md
│   └── testing-scenarios.md
└── .gitignore            # 包含 .pytest_cache/ *.bak 等规则
```

### 4.2 版本号管理
- 使用语义化版本：`MAJOR.MINOR.PATCH`
- README 版本号必须与 SKILL.md 中一致，与 Git tag 一致
- CHANGELOG 使用标准格式：`## [x.y.z] - YYYY-MM-DD`

### 4.3 品名一致性
- 中文品名、英文 README 标题、SKILL.md 标题三者必须一致
- 私有文件不得散落在根目录（应放入 references/）

## 5. 六种视觉主角类型

| 类型 | 适用场景 | 视觉特征 | 布局倾向 |
|------|---------|----------|---------|
| **数据可视化型** | 数据分析、业绩展示、市场研究 | 多图表、配色标注、数据标签 | TPL-DATA, TPL-CHART |
| **概念示意型** | 方案介绍、模型解释、方法论 | 流程图、架构图、概念图标 | TPL-CONCEPT, TPL-PROCESS |
| **大图故事型** | 案例展示、品牌故事、人设塑造 | 大幅背景图、少文字、留白多 | TPL-STORY, TPL-COVER |
| **对比拆分型** | 优劣势对比、方案选型、前后对照 | 左右分栏、表格、对照标注 | TPL-COMPARISON, TPL-TABLE |
| **流程叙事型** | 时间线、路线图、实施路径 | 时间轴、步骤编号、箭头 | TPL-TIMELINE, TPL-ROADMAP |
| **情感共鸣型** | 封面、结尾、核心观点强调 | 大字/留白/强视觉冲击 | TPL-HERO, TPL-QUOTE |

## 6. 五维度智能主题推荐

当需要选择品牌主题时，运行五维度评分引擎。

### 评分维度

| 维度 | 分值 | 说明 |
|------|------|------|
| 场景匹配 | 0-25 | "路演"内容自动加分给 pitch-vc 主题；"咨询报告"加给 consulting 主题 |
| 模板亲和 | 0-20 | 数据密集型报告不适合叙事型主题；叙事型内容不适合数据型主题 |
| 图表适配 | 0-25 | 从品牌预设 YAML 中提取每种主题对图表类型的偏好 |
| 信息密度 | 0-15 | 高密度内容匹配高密度主题（使用 `math.log1p()` 进行对数空间转换避免零除） |
| 视觉平衡 | 0-15 | 用对数空间比较避免零除问题 |

**满分 100 分。** 11 套品牌预设逐一打分，排出座次，给出推荐理由。

### 实现注意
- 使用 `math.log1p()` 将对数空间转换，避免纯数据报告场景下信息密度比计算出现除零错误
- 主题对象应使用 `getattr(theme, "inverse", False)` 而非 `theme.get("inverse")`——Theme 是类实例，不是 dict
- 模板分类列表必须与实际存在的模板名一致，不能包含不存在的模板名

## 7. 模板蓝图规范（Blueprint）

### 7.1 12 种布局家族

| 模板代码 | 名称 | 用途 |
|---------|------|------|
| TPL-COVER | 封面页 | 标题 + 副标题 + 品牌标识 |
| TPL-TOC | 目录页 | 议程 + 章节引导 |
| TPL-DATA | 数据页 | 数据图表 + 关键指标标注 |
| TPL-CHART | 图表页 | 单一大图展示 |
| TPL-CONCEPT | 概念页 | 方案+架构图+解释性文字 |
| TBL-PROCESS | 流程页 | 步骤编号+流程箭头+说明 |
| TPL-STORY | 故事页 | 大幅图片+文字标注 |
| TPL-COMPARISON | 对比页 | 左右分栏+对比项 |
| TPL-TABLE | 表格页 | 结构化数据表 |
| TPL-TIMELINE | 时间线页 | 时间轴+关键节点 |
| TPL-ROADMAP | 路线图页 | 多阶段路线+里程碑 |
| TPL-HERO | 强调页 | 大字标题+强视觉背景 |

### 7.2 Blueprint 格式规范

每页 Blueprint 必须包含：

```yaml
page:
  id: 1
  template: TPL-DATA          # 必须来自 12 种布局家族
  title: "市场规模与增长趋势"   # 页面标题
  subtitle: "2024-2028"       # 副标题（可选）
  content:
    type: chart               # content | chart | text | table | image
    spec:                     # 内容规范
      chart_type: bar         # 图表类型
      data_source: "research_briefing.md §3.1"
      data_keys: ["year", "revenue", "growth_rate"]
    items:                    # 对于 TPL-COMPARISON，items 必须是字典数组
      - label: "选项A"        # [{"label": "选项A"}, {"label": "选项B"}]
      - label: "选项B"        # 不能是字符串数组 ["选项A", "选项B"]
  theme: consulting           # 品牌主题（由推荐引擎选择）
  density: high               # low | medium | high
```

⚠️ **TPL-COMPARISON 格式注意：** 此模板的 items 必须是字典数组 `[{"label": "..."}, ...]`，不是字符串数组。

## 8. 图表类型规范

支持 16 种图表类型：

| 图表类型 | 适用数据 | 最佳场景 |
|---------|---------|---------|
| bar | 分类对比 | 市场份额、业绩对比 |
| column | 时间序列对比 | 年度数据对比 |
| line | 趋势展示 | 增长率、价格走势 |
| area | 累积趋势 | 市场规模累积 |
| stacked_bar | 组成对比 | 产品线构成对比 |
| stacked_column | 年度组成对比 | 收入构成 |
| pie | 占比展示 | 市场份额、预算分配 |
| donut | 占比展示(更现代) | 同上，视觉效果更佳 |
| radar | 多维度对比 | 能力评估、产品对比 |
| scatter | 相关性展示 | 投入产出分析 |
| bubble | 三维数据 | 市场规模+增长率+利润 |
| funnel | 漏斗转化 | 销售漏斗、用户转化 |
| waterfall | 累积变化 | 利润拆解、成本分析 |
| heatmap | 矩阵数据 | 风险矩阵、关联分析 |
| gauge | 达成率 | KPI 完成度、健康度 |
| treemap | 层级占比 | 业务板块构成 |

## 10. 常见错误与排查

| 错误 | 原因 | 解决 |
|------|------|------|
| `AttributeError: 'Theme' object has no attribute 'get'` | 把 Theme 类实例当 dict 访问 | 用 `getattr(theme, "inverse", False)` |
| `TPL-COMPARISON items mismatch` | items 是字符串数组而非字典数组 | 统一使用 `[{"label": "..."}]` 格式 |
| `ProcessPoolExecutor spawn loop` | Windows 缺少 `__main__` 保护 | 添加 `if __name__ == '__main__':` |
| `ImportError: cannot import name 'ChartRecommender'` | 函数式 API 被当作类导入 | 检查模块是函数式还是类式 API |
| `UnicodeDecodeError: 'gbk' codec...` | Windows 默认编码问题 | 显式指定 `encoding="utf-8"` |
| 模板分类匹配为空 | 分类列表中有不存在的模板名 | 同步分类列表与 12 种布局家族 |
